## 1. 产品与读者

| 项 | 说明 |
|----|------|
| 读者 | 使用者本人或小团队；在**手机邮件 App**与**桌面 Outlook / 网页邮箱**中阅读。 |
| 阅读场景 | 早间通勤、收盘后；光线可能较暗 → 需 **深色模式**友好。 |
| 内容特点 | **每日复盘**可达数万字；**周报**含多表、多段、智谱「风格诊断」；**成交通知**以表格式数据为主。 |
| 情绪预期 | **冷静、专业、可审计**（像对内风控纪要），而非面向散户的营销邮件。 |

## 2. 设计原则（可量化）

1. **首屏 3 秒内可读清**：系统是谁、什么日期/类型、是否成功。  
2. **纵向信息流 F 型**：标题左对齐为主；避免复杂多栏（邮件客户端易错位）。  
3. **对比度**：正文与背景对比度建议 **≥ 4.5:1**（正文灰字勿浅于 `#64748b` 类在浅底上）。  
4. **家族一致**：四类邮件**共用同一页头高度、同一页脚结构**；差异用 **副标题文案** + **可选 24px 内小标签**，不另做一套品牌。  
5. **长文不「墙」**：`h2` 间距、`p` 段间距、表格前后留白一致，避免密不透风。

---

## 3. 技术约束与邮件客户端现实

| 约束 | 详细说明 |
|------|----------|
| 模板引擎 | **Jinja2** 渲染 `email_base.html`，正文为 `content_html`（Markdown 转 HTML 或手写片段）。 |
| 最大宽度 | **600px** 是行业安全值；再宽在手机上会被缩放发糊。 |
| CSS 支持 | **嵌入式 `<style>`** 在 Apple Mail、多数 Webmail 有效；**Outlook Windows** 对 flex/grid/复杂选择器支持差 → **关键布局用 table 或简单 block + 内联 style** 更稳（当前实现以 div + 内联为主，设计时避免依赖 flex 多列正文）。 |
| 图片 | 周报通过 **`multipart/related` + `cid:`** 内嵌 PNG；必须 **`max-width:100%; height:auto`**，固定 `width` 慎用于大图。 |
| 纯文本 | 同封邮件必有 **text/plain** 备选；设计以 HTML 为准，plain 不要求版式一致。 |
| 关闭 HTML | 配置项 `email_html_template_enabled: false` 时可退回简单 HTML 壳；视觉规范仍以「开启模板」为主。 |

## 4. 版心与栅格

| 区域 | 建议值 | 说明 |
|------|--------|------|
| 外层背景 | 全宽 | 浅灰蓝 `#e8edf3`（浅色）/ 深色模式见 §5。 |
| 内容壳 `email-shell` | `max-width:600px`，水平居中 | 左右 padding **12～20px**（窄屏 **8～12px**）。 |
| 卡片 `email-card` | 同宽 | `border-radius: 12px`；`border:1px solid #e2e8f0`；轻阴影。 |
| 页头 `email-header` | 全宽于卡片内 | 上 padding **22～24px**，下 **20px**；与正文无额外白缝。 |
| 正文 `email-body` | 全宽于卡片内 | padding **24px 22px 28px**（移动端可 **18px 16px 22px**）。 |
| 页脚 `email-footer` | 全宽于卡片内 | 上边框分隔；padding **18px 22px**。 |

**竖向区块顺序（固定）**  
`[页头：系统名 + 日期]` → `[正文：动态 HTML]` → `[页脚：免责声明框 + 版权行]`。

## 7. 统一外壳：页头 / 正文壳 / 页脚

### 7.1 页头

- **必显**：`次日竞价半路复盘系统`（单行，不换行优先；过长可略缩字间距）。  
- **副标题**（变量 `header_date`）：示例  
  - 每日：`交易日 20260328`  
  - 周报：`2026年第12周 · 锚点 20260328`  
  - 模拟成交：`成交日 20260328`  
  - 权重异常：`2026 年第 12 周`  
- **视觉**：自上而下 **135° 线性渐变**（深蓝 → 中蓝 → 偏青深蓝）；**不要**在页头塞图标（部分客户端会拉伸）。

### 7.2 正文壳

- 白底卡片包一层 `<div class="content">`，内部才是 Markdown 或片段 HTML。  
- **不要**在模板里写死长文；设计稿用 **真实长度的占位段落**（至少 3 屏滚动）检验密度。

### 7.3 页脚

- **免责声明**（变量 `disclaimer`，可配置，默认）：  
  `以上内容基于公开数据与程序规则自动生成，仅供参考，不构成投资建议。股市有风险，投资需谨慎。`  
- **元信息**（变量 `footer_line`）：  
  `© 次日竞价半路复盘系统 · 版本 1.0`（版本来自 `email_app_version`）。

## 8. Markdown 正文元素与样式映射

程序侧使用 Python `markdown` 扩展：`extra`、`nl2br`、`fenced_code`、`tables`，可选 `codehilite`（需 Pygments）。

| Markdown / HTML | 期望视觉效果 |
|-----------------|---------------|
| `# 标题` → h1 | 最大号，深色，下划线或底边分隔，与页头区分明显 |
| `##` → h2 | 次级，紫/靛倾向（可与品牌一致） |
| `###` → h3 | 再次一级 |
| 段落 | 段间距 0.5～0.6em |
| `**粗体**` | 字重 600，颜色可略浅于纯黑 |
| 无序/有序列表 | 左缩进 1.35em，行距舒适 |
| 表格 | 全宽；表头灰蓝底；单元格左对齐为主，数字列可右对齐（若实现允许） |
| `>` 引用 | 左 4px 竖条 + 浅蓝背景 + 灰字 |
| 行内 `` `code` `` | 浅灰底、小圆角 |
| 围栏代码 | 深色底块、浅色字、圆角 8px、内边距 14～16px |
| `---` 分隔线 | 浅灰 1px，上下留白 |
| 图片（cid） | 块级居中，`max-width:100%`，圆角 8px |

## 9. 四类邮件：场景、主题、结构稿

### 9.1 每日复盘报告

| 项 | 内容 |
|----|------|
| 触发 | 夜间脚本或 Web `ReplayTask` 成功结束；失败亦可能发信。 |
| 主题（示例） | `【复盘】✅ {摘要一行} · {交易日}` / `【复盘】❌ 复盘失败 · {模式} · {日}` |
| 正文 | **完整 Markdown 报告**（市场摘要、龙头池、AI 章节等）。 |
| 页头副标题 | `extra_vars.header_date` → 如 `交易日 20260328` |
| 设计重点 | 长文层级、表格可读、代码块不溢出；失败时正文可能是纯错误栈，需 **等宽/红字**友好（可在 `email_notify` 层用另一片段，当前为同一壳）。 |

### 9.2 龙头池周度表现（周报）

| 项 | 内容 |
|----|------|
| 触发 | `scripts/weekly_performance_email.py`，配置 `enable_weekly_performance_email`。 |
| 主题（示例） | `【复盘】📊 龙头池周度表现 · {年}年第{周}周 · 锚点{YYYYMMDD}` |
| 正文 | Markdown：周统计、近四周、自然月、可选智谱「风格诊断」、可选模拟账户本周段。 |
| 附图 | 若开启 `weekly_email_attach_charts` 且文件存在：权重图 `weights_trend.png`、净值 `simulated_equity.png`，**接在正文 HTML 末尾**。 |
| 设计重点 | 文后大图区与正文间距；图下可加灰色说明「以上为程序生成示意图」。 |

### 9.3 模拟账户成交通知

| 项 | 内容 |
|----|------|
| 触发 | `enable_simulated_trade_notification`，买卖成交后守护线程发信。 |
| 主题（示例） | `【复盘】【模拟账户买入/卖出】{代码} {名称} {N}股@{价}` |
| 正文 | **非**大段 Markdown，而为 `build_simulated_trade_html` 生成的 **结构化 HTML 片段**，再套同一 `email_base`。 |
| 设计重点 | 徽章色与表格式对齐；数字千分位；提示框醒目但不刺眼。 |

### 9.4 策略权重异常提醒

| 项 | 内容 |
|----|------|
| 触发 | 周报流程内 `update_from_recent_returns` 返回 `weight_alerts`，且 `enable_weekly_weight_anomaly_email`。 |
| 主题（示例） | `【复盘】⚠️ 策略权重异常 · {年}-W{周} · {锚点}` |
| 正文 | 短 Markdown：`## 策略权重异常` + 列表项若干。 |
| 设计重点 | 与「成功复盘」绿勾情绪区分，用 **琥珀/橙** 左侧条或 warn 框（勿与免责声明黄框完全同色，可略偏橙）。 |

## 10. 模拟成交通知：字段级线框

自上而下块级顺序（设计稿按块标注间距 **16～20px**）：

1. **行 1：徽章**  
   - 文案：`买入` 或 `卖出`  
   - 样式：圆角胶囊；买 `#dcfce7/#166534`；卖 `#fee2e2/#991b1b`。

2. **块：成交明细**  
   - 标题：`成交明细`（h2 级，下划线分隔）  
   - 表：两列 **项目 | 内容**  
   - 行：股票代码（加粗）、名称、数量（股）、成交价（元）、成交金额（千分位）、理由（可能较长）、成交日。

3. **块：账户概况**  
   - 标题：`账户概况`  
   - 表：两列或键值对：总资产、现金、持仓市值、持仓只数、（可选）当日收益率、累计收益率（±%）。

4. **块：前三大持仓**  
   - 标题：`前三大持仓`  
   - 表：四列 **代码、名称、股数、市值≈**；无数据时单行「（无持仓）」。

5. **提示框 alert-info**  
   - 固定意图：模拟盘、可参考跟随节奏、实盘自负；**非投资建议**。

## 13. 无障碍与可读性

- **不要**仅用颜色区分买/卖：徽章内必有文字「买入/卖出」。  
- 链接（若有）需 **下划线或明显色**，勿仅依赖 hover。  
- 免责声明与正文对比度满足小号字可读。  
- 避免正文小于 **12px**（页脚 12px 已是下限）。

## 14. Stitch 交付物清单

1. **Frame A**：每日复盘 — 含长文滚动示意（可截 2～3 屏拼接说明）。

5. **色板页**：浅色 + 深色 HEX 表（见 §5）。

7. （可选）**组件导出**：页头 / 页脚 / 正文卡片分层 PNG/SVG。

## 15. 内容层级详述（与「摘要报表 / 每日复盘」类原型对齐）

> 本节供 **Stitch / 设计** 与 **产品** 对齐「邮件里到底有什么、先后顺序是什么」，避免只有版式没有内容规格。下列用语与章节名以**常见复盘长文结构**为例；实际以智谱输出与 `ReplayTask` 固定 prompt 为准，但层级逻辑一致。

### 15.1 页头（Header）

| 元素 | 说明 | 备注 |
|------|------|------|
| 左侧图标 | 古典建筑 / 机构感图形，白色，与金融场景一致 | 邮件中可用 Unicode 或内联小图，需测 Outlook |
| 主标题 | 如「次日竞价半路复盘系统」或产品化名称「次日竞价单摘要报表」 | 单行优先，过长可略缩字间距 |
| 副标题 / 元数据 | **生成日期**或**交易日**，格式建议 `YYYYMMDD` 或 `YYYY-MM-DD` | 与 `header_date` 变量对应 |
| 右侧图标 | 时钟、对勾、历史记录等「状态/时点」隐喻 | 非必须，增强「正式报告」感 |

页头背景为**深蓝渐变**，与正文白底形成强对比，读者第一眼识别「系统 + 日期」。

### 15.2 情报徽章（Intel Badge）与首屏摘要行

| 元素 | 说明 |
|------|------|
| 绿色胶囊英文标签 | 如 `INTERNAL INTELLIGENCE REPORT` 或原型中的 `BUILDING INTELLIGENCE REPORT`，小字号、大写、宽字距 | 表明「内部情报 / 程序生成」定位，非对外营销 |
| 紧跟其后的长标题行 | 常为主题行缩略 + **【摘要】** 一行三维度：`市场阶段：主升期/震荡期/退潮期`｜`适宜度：高/中/低`｜`置信度：高/中/低`，可附日期与「系统补全」说明 | 与推送标题、微信摘要同源逻辑 |

设计时徽章与首段摘要之间留白 **16～24px**，避免与正文黏连。

### 15.3 导读与「分类快讯」块（可选段落）

部分报告在长篇前设有**导读**，典型句式：

- 称呼：`尊敬的读者` + `[交易日 YYYYMMDD]`  
- 提示：`以下为快讯摘要，完整复盘见下文`  

正文前段可带 **分类标签**（用于扫读，非必须每封都有）：

| 标签 | 语义方向 |
|------|----------|
| 【要闻】 | 已公开、影响面较大的信息摘要 |
| 【传闻】 | 未证实或市场传言，需降权表述 |
| 【雷点】 | 业绩、监管、个股或板块风险点 |
| 【机会】 | 观察池、题材线索（仍非荐股） |
| 【策略】 | 与当前模式（次日竞价半路）相关的思路提示 |
| 【总结】 | 快讯段落的收束句 |

视觉：标签可用 **加粗** 或行首独立；标签与后续句子同一行或换行均可，但需**整段对齐方式一致**（建议左对齐）。

### 15.4 编号章节与「浅蓝摘要框」

长文主体多为 **Markdown 有序/无序标题**，常见顺序：

1. **市场阶段与情绪**（或合并为「市场前景与策略」）  
2. **主线与程序选股 / 领涨逻辑**  
3. **次日竞价预案 / 龙头池摘要**（常含表格）  
4. **风险与不适用场景**  

其中可在第一节前或节内重复出现 **【摘要】** 短行（与首行摘要可略有差异，侧重**本节结论**），置于**浅蓝左边框 + 浅底**的框内，与原型「浅蓝边框摘要框」一致。

### 15.5 典型子节应包含的信息类型（供表格化与抽检）

| 子节主题 | 建议出现的量化/结构化信息（示例） |
|----------|-----------------------------------|
| 市场情绪 | 涨停家数、跌停家数、炸板率、溢价、北向资金等（以程序数据为准） |
| 主线与龙头 | 程序给出的主线板块名、龙头池代码/名称/评分，必要时 AI 点评 |
| 次日预案 | 观察优先级、竞价关注点（连板、量比、换手、主力资金等关键词） |
| 风险 | 个股退市/ST、业绩窗口、模式不适用场景（如一字板无法半路等） |

### 15.6 表格：「次日竞价单摘要」类（高关注组件）

若正文含表格，常见列为：

| 列名 | 含义 | 视觉建议 |
|------|------|----------|
| 序号 | 列表顺序 | 窄列，居中 |
| 代码 | 6 位证券代码 | 等宽字体 |
| 名称 | 证券简称 | |
| 地位/角色 | 如核心、跟风、观察（程序或 AI 标签） | 可用浅色底区分 |
| 逻辑摘要 | 一句话理由 | 最长列，自动换行 |

表头行：**浅灰蓝底 + 小号大写字母或加粗**，与数据行区分。涨跌幅若在表内，可用**绿/红药丸**（需 HTML 片段或后处理，纯 Markdown 表格不自带）。

### 15.7 风险与免责声明（正文内 + 页脚双层）

- **正文末**：单独小节「风险与不适用场景」，条列式最佳。  
- **页脚黄框**：固定或模板化免责全文（不构成投资建议）。  
- **黄框内或下**：可再叠一层 **算法生成** 提示（与原型「浅黄警示框 + 警告图标」一致）。  
- **水印**：大号浅色 `NB` 等字母水印，仅装饰，不参与信息层级。

### 15.8 页脚英文链接行与版权

原型常见第三行信息：

- 大写英文链接：`CONFIDENTIALITY` / `SYSTEM STATUS` / `ARCHIVE`（或简写为 Confidentiality · System status · Archive）  
- 英文版权：`© {YEAR} NEXT-DAY BIDDING SYSTEM | INTERNAL INTELLIGENCE REPORT`  
- 与中文「© 次日竞价半路复盘系统 · 版本 x.x」**并存**，服务不同阅读习惯的客户端预览。

### 15.9 与程序实际输出的关系（Markdown → HTML）

| 原型块 | 程序侧常见来源 |
|--------|----------------|
| 全文 | 智谱返回的 Markdown，经 `markdown_to_html` |
| 标题层级 | `##` `###` 对应 h2/h3；样式在 `email_base.html` 的 `.content` 中定义 |
| 表格 | GFM 表格渲染为 `<table>` |
| 代码块 | 围栏代码 → `<pre><code>`，JSON/参数块多为深色底 |
| 引用 | `>` → `<blockquote>`，适合放「风格诊断」摘录 |
| 页头/页脚/徽章/水印 | Jinja2 模板固定，**不在**模型正文内 |

若模型未输出某节，邮件仍应有统一外壳；**不得**在模板中写死具体股票结论。
