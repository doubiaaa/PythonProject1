# Stitch 设计简报：次日竞价半路复盘系统

> **用途**：将本文**全文或分节**粘贴到 [Google Stitch](https://stitch.withgoogle.com/)（或同类 AI 界面原型工具）的提示中，生成**可评审的界面原型**；定稿后由开发对照修改 `templates/index.html`（及少量静态资源）。  
> **详细架构**：见仓库根目录 `ARCHITECTURE.md`。

---

## 1. 产品一句话

**程序算清量，模型写得像。**

- **程序**：A 股交易日历、行情与规则、龙头池打分与标签、本地持久化、（可选）模拟撮合与策略权重更新。  
- **模型**：DeepSeek 在固定章节结构下生成长文 Markdown 复盘，**不替代**程序数据与风控责任。  
- **合规**：输出仅供研究参考，**不构成投资建议**；界面与邮件须有免责声明气质（克制、专业、非喊单）。

---

## 2. 系统边界（给设计师的约束）

| 要做 | 不做 |
|------|------|
| 收盘后 Web 触发单次复盘、展示 Markdown 结果 | 不提供实盘下单 |
| 可选：微信（Server酱）、邮件推送报告 | 不保证收益、不担保第三方行情实时性 |
| 可选：模拟账户、周报脚本、权重图（多为**脚本/邮件**，非本页核心） | GitHub Actions 不持久化本地 `data/`，勿在原型中假设「云端自动同步账户文件」 |

---

## 3. 技术现实（原型需可落地）

| 项 | 说明 |
|----|------|
| 前端 | **单页** Flask 模板：`templates/index.html`，**无** React/Vue 工程；样式为 **Tailwind CDN** + 少量自定义 class。 |
| 交互数据 | `POST /api/start_replay`、`GET /api/task_status`（轮询）、`GET /api/get_defaults`。 |
| 长文展示 | 浏览器端 **marked + DOMPurify** 渲染 Markdown；失败时有简易分段回退。 |
| 品牌 | 中文为主；标题常见：**次日竞价半路模式** / **次日竞价半路复盘系统**。 |

**Stitch 产出物建议**：先出 **Desktop 1440** 与 **Mobile 390** 两套 Frame；标注与现有区块对应关系（见 §5），便于开发逐块替换 HTML/CSS。

## 4. 逻辑架构（简化，供理解信息从哪里来）

```text
用户浏览器 → Flask(/, /api/*)
         → ReplayTask（单次复盘）
         → DataFetcher + 竞价半路策略（龙头池、市场摘要）
         → DeepSeek（长文）
         → watchlist 持久化、（可选）模拟账户、Server酱、SMTP 邮件
```

并行入口（**不必画在同一页**，若要做「信息架构图」可提及）：`nightly_replay.py`、`weekly_performance_email.py`、`simulated_morning_buy.py`。

## 5. 当前 Web 单页结构（请在此基础上优化，而非从零虚构业务）

页面为**纵向信息流**，从上到下建议对应关系：

| 顺序 | 区块名 | 功能与内容 |
|------|--------|------------|
| A | **Hero** | 产品名、一句话价值、合规小字（数据来源说明） |
| B | **配置** | 复盘日期（YYYYMMDD）、今日/昨日；DeepSeek API Key（可显隐）；SMTP 区块（主机、端口、用户、密码、发件人、收件人、SSL 勾选） |
| C | **情绪仪表盘**（四格） | 情绪温度、市场阶段、建议仓位、北向资金——**当前为占位**，成功生成报告后由前端脚本从正文中**正则抽取**更新（原型需预留「空态 / 已填充态」） |
| D | **主操作** | 主按钮「生成复盘报告」；状态文案、进度条、百分比、耗时；说明文案（生成中锁定按钮等） |
| E | **实时日志** | 等宽字体、可滚动、可清空 |
| F | **复盘结果** | Markdown 渲染区（大段正文）、复制全文、保存为 txt、清空；失败时为错误样式 |
| G | **页脚** | 简短免责声明 |

**关键交互**：仅允许**一个**复盘任务并发（409 时提示「已有任务」）；运行中轮询 `task_status` 拿 `logs`、`progress`、`status`、`result`。

## 6. API 契约（原型中的按钮/状态应对齐）

### `POST /api/start_replay`

JSON 字段示例：`date`、`api_key`、`serverchan_sendkey`、`smtp_host`、`smtp_port`、`smtp_user`、`smtp_password`、`smtp_from`、`mail_to`、`smtp_ssl`。  
成功：`{ "status": "started" }`；冲突：`409` + `{ "error": "..." }`。

### `GET /api/task_status`

返回含：`status`（如 `idle` / `running` / `completed` / `error`）、`result`、`logs`（数组）、`progress`（0–100 可选）。

### `GET /api/get_defaults`

预填 Key、Server酱、SMTP 等（密码不回显，需用户重填）。

## 7. 给 Stitch 的「设计任务」提示（可直接粘贴）

请用中文思考与标注。

1. **风格**：专业金融科技向；默认可延续**深色渐变 + 玻璃拟态 + 靛蓝/青色点缀**（与现站一致），或提供一套**浅色专业版**作为备选 Frame。  
2. **信息层级**：配置区表单分组清晰；主 CTA 醒目但不「传销感」；结果区阅读舒适（行长、标题层级、代码块）。  
3. **响应式**：手机端配置与 SMTP 改为合理折行，结果区最小高度与滚动明确。  
4. **状态**：为「就绪 / 运行中 / 成功 / 失败」提供明确视觉；日志区与结果区分明。  
5. **无障碍**：主按钮、`aria-busy`、焦点态可略提（不必细画）。  
6. **不要**增加需要后端新接口的复杂功能（如「历史报告列表」）除非标注为**未来迭代**。

## 8. 定稿后：如何按原型改 UI（开发流程）

1. 在 Stitch 导出 **标注截图** 或 **设计说明**（间距、圆角、色板、字号）。  
2. 打开 `templates/index.html`，按区块 A–G 对照改：  
   - Tailwind class、`glass-panel`/`mesh-bg` 等自定义类；  
   - 勿改 API 路径与 JS 里 `getElementById` 的 **id**（除非同步改 `bindEvents` / 轮询逻辑）。  
3. 若改配色：同步检查 `.report-md` 下 Markdown 标题、表格、代码块对比度。  
4. 本地 `python run.py` 手测：启动复盘、轮询、结果渲染、复制与保存。
